# Task: Logging and request-ID middleware

## Description

Create `app/core/logging.py` (structured-JSON `structlog` setup) and `app/core/middleware.py` (`RequestIDMiddleware`). These two modules form the cross-cutting branch of the DAG — they depend only on the scaffold and unblock the app factory (008), running in parallel with the schemas/service/routers branch. Implements PRD §NFR (structured logging) and PRD §FR-7 (request-ID propagation).

## Acceptance Criteria

### Logging (`app/core/logging.py`)

- [ ] Exposes `configure_logging() -> None` that idempotently configures `structlog` with:
  - JSON renderer (final processor).
  - ISO-8601 UTC timestamps.
  - Standard processors: `add_log_level`, `TimeStamper(fmt="iso", utc=True)`, `format_exc_info`, `JSONRenderer`.
- [ ] Exposes a helper to bind a `request_id` value to the per-request log context (`structlog.contextvars.bind_contextvars(request_id=...)` style), and a clear/unbind on request completion.
- [ ] No direct `print` calls; no `logging.basicConfig` overrides outside of `configure_logging`.

### Middleware (`app/core/middleware.py`)

- [ ] `RequestIDMiddleware` is a Starlette `BaseHTTPMiddleware` subclass.
- [ ] On request: read `X-Request-ID` header; if absent, generate `uuid.uuid4().hex` (canonical lowercase 32-char hex).
- [ ] Bind the chosen `request_id` to log context for the duration of the request; clear on response (use try/finally).
- [ ] Set `X-Request-ID` on every outgoing response (success or error).

### Gate (mirrors Phases C and G)

- [ ] `configure_logging()` produces parseable JSON when emitting a test log line.
- [ ] `RequestIDMiddleware` instantiates and its `dispatch` method delegates to the next handler without error in a unit-level smoke test.

## Technical Details

### Files to create

- `app/core/logging.py`
- `app/core/middleware.py`

### Notes

- The middleware sits in `app/core/`, not `app/middleware/`, because there is exactly one middleware. ADR-003 favours the simplest layout that conveys intent.
- The middleware MUST set the response header even for handler-raised exceptions; use try/finally around `await call_next(request)`.
